# 交互式可视化 · 规范手册

本项目每一关除了 Python 代码,还配一套**交互式 HTML 可视化**(先建直觉,再读代码)。
本文件是这些页面的**规范手册** —— 新增章节、改样式前先读这里,保持全站一致。

> 写给未来的自己(和 AI 助手):别忘了这些约定。

## 目录结构

**所有 HTML 都放在 `web/` 目录下同一层**(扁平,不再嵌套);**文件名一律英文**:

```
web/
  index.html                     # 唯一首页(总入口),扁平列出所有章节
  glossary.html                  # 名词表(术语字典)
  notes.html                     # 学习札记 / 彩蛋:正课之外的小故事(可持续追加 article)
  01_bigram_crossentropy_viz.html  # 第 1 章可视化
  02_attention_viz.html          # 第 2 章可视化
  03_transformer_viz.html        # 第 3 章可视化(完整 Transformer)
  04_gpt_machine_viz.html        # 整机俯瞰 · 全景收尾(装机 + 跑起来)
可视化规范.md                     # 本文件(在项目根)
```

约定:
- HTML 全在 `web/` 同层,页面互跳就是 `./xxx.html`,最简单。
- 文件名只用英文(中文标题放页面内容里,别放进文件名)。
- 首页**只有一层**:`index.html` 直接列到具体章节,不搞"总入口→阶段索引→章节"多级下钻。
- Python 代码按 phase 分目录(`phase1-nanogpt/` 等),跟 `web/` 平级。

## 图形优先:这是图解,不是笔记(最高优先级)

这些页面本质是**图形化教学**,不是文档。一句话标准:**一屏里"能看图 / 能交互"的部分,要明显多过纯文字。**
文字是给图补一句注脚,不是主体。

- **一步只讲一条主线**:配一个能亲手拨的交互 + 一两句短说明。说明只点透"这在干嘛 / 为什么",不展开论证。
- **能用图或交互表达的,就别写成文字。** 先问"这能不能画成图、做成可拨的交互",不能再考虑文字。
- **深问题用渐进展开,别堆 note 墙。** 用户问的细节答案要留在页里(承诺过),但**默认收起** ——
  放进 `<details class="dig"><summary>一行标题</summary><div class="dig-body">…</div></details>`,点开才显示。
  主线保持干净,想深入的人自己点。标题要像"问题"不像"小节名"。
- **🔒 硬规则:凡"带问题"的解释,一律折叠,绝不平铺。** 只要一段内容的标题/开头是一个<b>问题</b>
  (「为什么…?」「…吗?」「…是什么?」「别混 / 别误会…」这类答疑、辨析、纠错),
  <b>必须</b>用 `<details class="dig">`,<b>不准</b>写成平铺的 `.note`。
  `.note` 只留给两种:① 交互的动态/静态<b>一句话</b>说明;② 一句话<b>结论 / takeaway</b>。
  —— 这是用户反复强调并拍板的规则:答疑式内容只要平铺,页面立刻就乱。写完自检:每个 panel 里
  带「?」的段落是不是都在 `.dig` 里?是平铺 `.note` 就是 bug。
  (`.dig` 个数不设硬上限:有几个问题就几个折叠,但每个标题必须是一个真问题。)
- **记答案 ≠ 堆正文。** 一个问题的答案,优先级:① 变成图/交互 → ② 一句话融进主线 → ③ 收进 `.dig` → ④ 干脆不写。
  绝不是默认全部摊成 `.note` 正文。
- **🔒 正面陈述,别把"某个人的具体猜错"写成纠错。** 用户某次问话里的个人误读(如「是不是 3 个地方各取 3 个 token」),
  是**他一个人**的思路,别的读者未必这么想 —— **不要**把它写成「不是 X,而是 Y」的反驳段落塞进页面。
  **直接、正常地把事情讲对就行**,不替未来读者预设他们会犯同一个错(不"举一反三"、不过度预设)。
  `.dig` 标题可以是个自然问题,但正文**正面说明**,不针对某个具体错误猜测做辩驳。
  (例外:像「block_size vs batch_size 撞脸」「两个 softmax」这类**普遍**的混淆点,才值得显式点出「别混」。)
- **🔒 网页只放"自包含、客观"的教学内容,不是"对用户的回答"。** 用户提问 → **先在对话里回答他**;
  网页**不是**用来粘贴回答的地方。只有当某段内容对<b>任何</b>读者都有教学价值时,才写进网页,而且要写得
  **不看用户的提问也能读懂**:① 不引用"我之前说过的话 / 用户的原话"(如『"向量缩到 2 维"说的是…』——
  读者根本没见过这句,只会懵);② 只描述<b>页面上看得到的东西</b>和客观事实;③ 读者看不到聊天里的问题,
  所以正文不能默认"你刚才问的那个"。**判据:把这段话单独拎给一个没看过对话的人,他能不能看懂?不能就是写错了。**
- **反面教训(本页真踩过):** 为了"把每个问题都写进去",在 02 页 Step 3/4/5 堆了七八个 `.note`,
  变成八股笔记墙,用户直接反馈"读不下去"。整改方式就是上面的渐进展开。

> 改完自检:每个 panel 滚一遍 —— 主线是不是图/交互打头?纯文字段落是不是都短、或已收进 `.dig`?

## 页面铁律

1. **单文件、零依赖、离线可开**:不引外部 JS/CSS/字体/CDN,纯 vanilla JS,双击就能跑。
2. **每个章节页左上角必须有 `← 返回首页`**(`.home-link` → `index.html`)。
3. 顶部有 `.filename` 角标显示自己的文件名。
4. 语言简体中文;术语保留英文原词(softmax、logits…)。
5. **首页是子页的镜像 —— 改子页必须同步首页。** 任何时候改了某章节页的 **step 标题 / 数量**、
   或**章节描述**,**必须同步更新 `index.html`** 里该章卡片的:描述、`.steps` 步骤列表(文字 +
   `#step-N` 深链)、"共 N 步"。两边对不上就是 bug。改完自检:首页每个 step 链接点进去标题一致。

## 章节页结构(以 01 / 02 为模板)

分步导航(step-tab),五步左右一关,逐步推进:

- 顶部:`.home-link` → `.filename` → `<h1>` → `.subtitle`。
- `.steps-nav`:一排 `.step-tab`(`data-step="0..N"`),点击切换,当前步加 `.active`。
- **超参速查条 `.hp-bar`**:step-nav 下方**常驻一行**,用颜色标签列出本章关键超参
  (batch_size / block_size / n_embd / head_size / vocab_size…),每个带 `值 + 一句话含义`,
  让用户**每一步都能看到那些数字是什么**。撞脸的数值(如多个 32)必须注明"含义不同"。
- **占位值要提示**:演示里为了好画用的小数字(如向量取 2 维、`torch.arange` 填的 0–5)
  必须明确标注"只是占位 / 为了上屏",并给出真实代码里对应什么(如 head_size=32、真实 value 向量)。
- **数字永远带名字,不说裸数值。** 多个超参同值时尤其要命(本项目 batch_size / n_embd / head_size 都是 32)。
  正文写 `n_embd=32`、`head_size=32`,绝不只写"32",否则读者会把"维度"误当成"句子数"。
- **维度色标组件 `.d`(代码不动,只靠标签 —— 用户拍板的方案)。** 凡是出现维度数字(算式里的 `32×32`、
  形状 `65×32`)都套 `.d` 色标:每个超参一个固定颜色 + 内联小名字 `<i>`,颜色与 `.hp-bar` 一一对应
  (hp-bar 就是图例)。**绝不让维度数字裸奔**。五个类名:
  ```html
  <span class="d d-batch">32<i>batch</i></span>   <!-- batch_size #b8512f/#F7E6DE -->
  <span class="d d-block">8<i>block</i></span>    <!-- block_size/T  #3f7d63/#E4EFE9 -->
  <span class="d d-emb">32<i>n_embd</i></span>    <!-- n_embd/C       #3a63c0/#E5EBF9 -->
  <span class="d d-head">32<i>head</i></span>     <!-- head_size      #a3548c/#F6E6F0 -->
  <span class="d d-vocab">65<i>vocab</i></span>   <!-- vocab_size     #a87b1f/#FAEFD6 -->
  ```
  CSS 见 02 页 `:root` 下 `.d` 段,新页面整段复制。**不改 `.py` 真实超参**(保持 Karpathy 一致),
  只靠颜色+名字消歧。
- 每步一个 `.panel`(`id="panel0..N"`),当前步加 `.active` 显示。
- 底部 `.nav`:`← 上一步` / `下一步 →`;最后一步按钮变 `完成 🎉` 触发庆祝。
- **深链**:URL hash `#step-N`(N 从 1 起)。`stepFromHash()` 解析 + `goto()` 切换,用
  `history.replaceState` 同步 hash。首页可用 `xxx.html#step-3` 直接深链到某一步。
- **通关庆祝**:最后一步点"完成" → `celebrate()`(横幅 `.congrats` + canvas confetti)。

### 叙事要求(最重要)
**像第 1 章那样真正一步步推进**,别把一页堆成名词解释。每步结尾用 `↳ 下一步:…`(`.next-hint`)
为下一步埋动机,让"下一步"是被前一步逼出来的。每步配一个**能亲手拨动的交互**(点序列、
拖滑块、逐步点按钮),先给直觉再落到代码。一关讲清一条主线。

## 按钮规则(统一)

- **亮(橙色填充 `.btn.primary`)= 当前激活 / 推荐操作**;不亮(`.btn` 或 `.btn.ghost`)= 默认 / 次要。
- **同一组"逐步演示按钮":点谁谁高亮,其余中性,样式必须一致且都有 active 态。** 统一辅助函数:
  ```js
  function setActiveBtn(containerId, n){
    document.querySelectorAll('#'+containerId+' .btn').forEach((b,i)=>
      b.classList.toggle('primary', i===n));
  }
  ```
  ⚠️ 不要同组里某个长期 `ghost`/透明、别的不是 —— 这是之前犯过的不一致 bug。

## 名词解释:气泡 + 名词表(全站统一组件)

形式 = **内联 hover 气泡** + **集中的 `glossary.html`**。

### 正文里加术语
```html
<span class="term"
      data-term="下三角 · lower-triangular"   <!-- 气泡标题 -->
      data-key="tril"                          <!-- glossary.html 锚点 -->
      data-def="对角线及左下保留,右上全 0…">  <!-- 一句话定义 -->
  下三角
</span>
```
- 效果:虚线下划线;hover/点击弹气泡(标题 + 定义 + `更多 → 名词表 ↗` 链到 `glossary.html#<data-key>`)。
- 组件 = 一段 CSS(`.term` / `#termTip`)+ 一个 IIFE(建浮层、绑定 `.term`)。**新页面整段复制**这两块。
- ⚠️ 气泡 JS 在**加载时一次性**绑定 `.term`。**JS 动态生成**的文字(如热力图说明)里的术语不会有气泡 ——
  要解释的术语放在**静态 HTML** 里。

## 引用代码:用「文件 + 函数/类名」,别用行号

页面里引用 `.py` 代码时,**只写到文件 + 函数/类名**(如 `02_attention.py 的 forward`),
**绝不写固定行号**(`02_attention.py:128`)。行号一改代码就失效,维护成本高、容易对不上。

## 札记页 `notes.html`(彩蛋 / 边角料)

正课之外的小故事(历史、八卦、前世今生)放这里,**不塞进章节页**避免喧宾夺主。约定:

- 复用全站骨架:暖色变量 / `.home-link` / `.filename` / 名词气泡组件(整段复制),但**不需要 step-tab**。
- 每条彩蛋 = 一个 `<article class="note-card" id="egg-xxx">`,顶部一个 `.egg-tag`(如「彩蛋 01 · 前世今生」),
  可深链 `notes.html#egg-xxx`。新彩蛋往下追加 article 即可。
- **双向挂钩**:首页 header/footer 有入口;相关 `glossary.html` 词条末尾用一行 🥚 链到对应 `#egg-xxx`。

## 新增一章的清单

1. 在 `web/` 下建 `0X_xxx_viz.html`(英文名),复制 01/02 骨架(变量 / home-link / step-tab / nav / 气泡组件 / 庆祝)。
2. 设计 5 步左右递进叙事,每步一个交互 + `↳ 下一步` 钩子。
3. 新术语:正文加 `.term`,并在 `glossary.html` 补词条(同步 TOC)。
4. `index.html` 对应 phase 分组里加章节卡(含 `#step-N` 深链步骤列表),更新锁定/进度状态。
5. 自检:返回首页可用、`#step-N` 深链可用、同组按钮有 active 态、气泡能弹、离线双击能开。
