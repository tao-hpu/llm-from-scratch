## 一、三个文件的进化路线

每一步只解决上一步的一个短板，数据管线和训练循环几乎不动，**变的只有中间那个模型**。

| 文件 | 模型 | 能看多远 | val loss | 短板 → 下一步解决 |
|---|---|---|---|---|
| `01_bigram.py` | 一张 65×65 查找表 | 前 1 个字符 | 2.49 | 只有 1 字记忆 → 加注意力 |
| `02_attention.py` | 单头自注意力 | 前 8 个字符 | 2.41 | 单头、单层、太浅 → 加宽加深 |
| `03_transformer.py` | 完整 Transformer | 前 64 个字符 | **1.57** | 真·GPT，Phase 1 终点 |

**核心一句话**：语言模型 = 「给定前文，在词表上输出下一个 token 的概率分布」的分类器。
从 bigram 到 GPT，变的只是「用多少上下文、用多大容量」去算这个分布。

### 01 → 02 加了什么：自注意力
- bigram 只查「前一个字符」对应的那一行。
- 自注意力让第 t 个位置能**往回看 0..t 的所有位置**，并且「谁和我相关，我就多看谁」（权重由 Q·K 点积算出来，不是固定平均）。
- 因果掩码（下三角）保证「只看过去，不看未来」。

### 02 → 03 加了四样东西（质变来源）
| 组件 | 一句话作用 | 不要它会怎样 |
|---|---|---|
| **Multi-Head** | 多个头并行，各看不同模式（语法/主谓/标点） | 只有一种注意力视角，表达力弱 |
| **FeedForward** | 逐位置的小 MLP，注意力负责「沟通」，它负责「思考」 | 只有信息搬运，没有逐位置加工 |
| **残差连接** `x = x + sublayer(x)` | 给梯度一条直通底层的「高速公路」 | 堆深了梯度消失，根本训不动 |
| **LayerNorm** | 稳定每层激活分布 | 深网络训练发散 |

> 残差 + LayerNorm 是「能不能堆深」的关键。你以后研究优化器/LoRA 时会反复碰到它们——
> 深层网络为什么训得动，一半答案在这两个东西里。

## 二、训练脚本的「7 大步骤」骨架

是的，这就是训练任何模型前最重要的大步骤。**`01/02/03` 乃至以后的 SFT/LoRA，骨架完全一样**，
只是第 1 步换数据、第 4 步换模型。记住这 7 步，你就再也不会对着训练脚本发懵：

```
1. 准备数据      读原始文本/语料
2. Tokenize     文本 ↔ 整数（让模型能"吃"）         encode / decode
3. 切 batch      切成 (x=输入, y=目标=下一个token)    get_batch
4. 定义模型      forward: token → 每个位置对"下一个token"的得分(logits)
5. 定义 loss     cross_entropy(预测分布, 真实下一个token)
6. 训练循环      zero_grad → backward → step  反复：预测→算错→改参数
7. 评估 + 采样   estimate_loss 监控曲线；generate 让模型"说话"
```

**input / output 是什么**：
- 模型的 input = 一批 token 序列 `(B, T)`，每个数字是一个字符的编号。
- 模型的 output = `(B, T, vocab_size)`，即**每个位置上、对「下一个字符是谁」的 65 个得分**。
- loss = 这个预测分布和「真实下一个字符」差多远。训练就是让这个差越来越小。

> 🔑 **对你最重要的一句**：以后做 SFT/LoRA，就是把第 1 步换成「对话数据」、第 4 步换成
> 「加载预训练模型 + 插 LoRA」。你过去两次 SFT 失败，**不是骨架坏了，是某个旋钮拨错了**
> （最可能是：数据格式/chat template、学习率、或者根本没做第 7 步的评估）。

## 三、调参手册（旋钮表）

`03_transformer.py` 顶部那堆超参数，就是你「能拨的按钮」。下面是每个的说明书。

### 旋钮总表

| 旋钮 | 当前值 | 管什么 | 拨大 → | 拨小 → | 好的区间 | 什么时候动它 |
|---|---|---|---|---|---|---|
| **lr** 学习率 | 1e-3 | 每步参数更新的步长（**最敏感**） | 太大→loss 爆炸/NaN | 太小→收敛慢/卡住 | 小模型 1e-3；大模型 3e-4 | **第一个该调的**。发散就降，不动就升 |
| **n_embd** 宽度 | 128 | 模型容量（每个 token 的向量维度） | 容量↑、更慢、过拟合风险↑ | 容量↓、欠拟合 | 128–384（小模型） | 欠拟合时升 |
| **n_layer** 深度 | 4 | 模型容量（堆几层 Block） | 抽象力↑、更难训 | 容量↓ | 4–8（小模型） | 欠拟合时升（靠残差+LN 才堆得动） |
| **n_head** 头数 | 4 | 注意力头数，head_size = n_embd/n_head | 视角更多样、每头更小 | 视角少、每头更大 | 让 head_size 落在 32–64 | 跟着 n_embd 定，**必须整除** |
| **block_size** 上下文 | 64 | 能往回看多远 | 看更远，但计算 O(T²) 暴涨 | 看不远 | 64–256（字符级） | 任务需要长依赖时升 |
| **batch_size** | 64 | 一步看多少条序列 | 梯度更稳更快，吃显存 | 梯度更噪 | 显存允许下尽量大 | 训练抖就升；OOM 就降 |
| **dropout** | 0.1 | 正则强度（随机丢连接） | 抗过拟合↑，太高反而欠拟合 | 几乎不正则 | 0.0–0.3 | **train<<val 时升** |
| **max_iters** | 5000 | 训多久 | 更充分，但可能过拟合 | 欠训 | 看 val 何时不再降 | val 不降就停（早停） |

> `eval_interval` / `eval_iters` 不是模型旋钮，只是「多久看一次曲线、每次看多少 batch」，
> 影响监控精度和打印频率，不影响模型本身。

### 决策树：看 train/val 两条曲线，对症下药

诊断永远从**对比 train loss 和 val loss** 开始：

```
两条都高、且贴在一起        →  欠拟合（容量不够）
                            →  升 n_embd / n_layer / max_iters

train 低、val 高、差距拉大    →  过拟合（背答案了）
                            →  升 dropout / 减小模型 / 加数据 / 早停

loss 变 NaN 或上下乱跳        →  lr 太大
                            →  降 lr（先降 3 倍试试）

loss 几乎不降               →  lr 太小，或数据/代码有错
                            →  升 lr；检查 get_batch 的 x/y 对没对齐

跑太慢                      →  降 block_size / batch_size（牺牲质量）
                            →  或换上 GPU（4090 / A100）
```

### 拨钮的优先级（不知道先调啥时）
1. **lr**——最敏感，先把它调到「loss 稳定下降不发散」。
2. **模型容量（n_embd / n_layer）**——欠拟合就加，直到 val 开始和 train 分开。
3. **dropout / max_iters**——出现过拟合后再用它们收尾。
4. block_size / batch_size——多数时候按机器能力定死，不常动。

## 四、Mac 版 vs 4090 版

`03` 顶部注释里给了两套配置：

| 配置 | n_embd | n_layer | n_head | block_size | dropout | 结果 | 在哪跑 |
|---|---|---|---|---|---|---|---|
| Mac 版（当前） | 128 | 4 | 4 | 64 | 0.1 | val ~1.57，约 8 分钟 | M4 Max（MPS） |
| 4090 版 | 384 | 6 | 6 | 256 | 0.2 | val ~1.48，更像样 | 4090 |

学习阶段在 Mac 上把机制跑通就够了；想要「真·能读的莎士比亚」，把 4090 版配置丢到 4090 上跑。
（这正对应 README 的硬件映射：小规模在 Mac，scale 才上 GPU。）
